# Система аналізу та візуалізації продуктивності алгоритмів

**Лабораторна робота №2** з дисципліни "Об'єктно-орієнтоване проєктування".
**Автор:** Давиденко В. О., група Іпс-21.

##  Опис проєкту
Цей проєкт є консольним застосунком для тестування, аналізу та візуалізації продуктивності складних структур даних та алгоритмів. Основна мета проєкту — практичне застосування класичних патернів проєктування (GoF) в умовах, наближених до реальної розробки.

Система генерує тестові дані, виконує алгоритми із заміром часу та зберігає результати у вигляді згенерованого HTML-звіту (з використанням Bootstrap 5).

### Реалізовані алгоритми:
1. **Red-Black Tree (Order Statistic Tree)** — вставка, балансування, пошук за рангом (`OsSelect`, `OsRank`).
2. **B-Tree** — вставка з імітацією роботи віртуального диска та розбиттям вузлів.
3. **Multiway Merge Sort** — багатошляхове сортування злиттям з використанням `PriorityQueue` (мінімальної купи).

##  Архітектура та Патерни Проєктування

У проєкті реалізовано патерни з усіх трьох обов'язкових категорій (Твірні, Структурні, Поведінкові), що забезпечує гнучкість, розширюваність та чистоту коду.

### Поведінкові патерни (Behavioral)
* **Strategy (`IAlgorithmStrategy`)**: Визначає загальний інтерфейс для всіх алгоритмів. Дозволяє динамічно вибирати алгоритм для виконання під час роботи програми без зміни клієнтського коду.
* **Template Method (`AlgorithmTemplate`)**: Базовий абстрактний клас, який задає загальний скелет виконання (перевірка на порожні дані, ініціалізація), делегуючи саму логіку алгоритму (метод `PerformAlgorithm`) класам-нащадкам.
* **Command (`ICommand`, `RunAlgorithmCommand`, `SaveReportCommand`)**: Інкапсулює запити користувача (пункти меню) в окремі об'єкти. Це дозволило створити гнучкий інтерактивний інтерфейс (CLI), який легко розширювати новими функціями.

### Структурні патерни (Structural)
* **Decorator (`TimerDecorator`)**: Обгортає будь-який об'єкт типу `IAlgorithmStrategy` для вимірювання практичного часу його виконання. Додає новий функціонал (таймер), не втручаючись у вихідний код самих алгоритмів (дотримання принципу Open/Closed).
* **Facade (`AlgorithmFacade`)**: Надає спрощений інтерфейс до складної підсистеми класів. Фасад бере на себе управління життєвим циклом виконання: він сам звертається до Фабрики, обгортає алгоритм у Декоратор, запускає його і відправляє результати у Singleton. Це повністю розвантажило клієнтський код від зайвої складності.
### Твірні патерни (Creational)
* **Factory Method (`AlgorithmFactory`)**: Централізовано створює екземпляри алгоритмів за текстовим ключем. Ізолює клієнтський код від процесу інстанціювання конкретних класів алгоритмів.
* **Singleton (`HtmlReportExporter`)**: Гарантує, що у системі існує лише один екземпляр генератора звітів. Відповідає за накопичення метрик під час роботи програми та фінальну генерацію `.html` файлу з результатами.

##  Структура репозиторію

- `src/Core/` — Бізнес-логіка, патерни та алгоритми (незалежна бібліотека).
- `src/App/` — Клієнтський код, CLI інтерфейс (патерн Command).
- `tests/Core.Tests/` — NUnit тести для перевірки інваріантів та логіки алгоритмів.
- `html/` — Автоматично згенерована документація Doxygen (UML та опис класів).
- `PerformanceReport.html` — Згенерований застосунком звіт (з'являється після виконання програми).
